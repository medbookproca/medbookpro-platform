# Database specification

This is a proposed schema contract, not executable SQL. All primary keys are UUIDs generated by the database in a future migration unless stated otherwise. All timestamps are UTC `timestamptz`. Every table must have `created_at` and, where mutable, `updated_at`; actor and audit columns are added where material.

## Shared conventions

- Status values use constrained text or an enum only after migration review.
- Names and email addresses have bounded lengths; normalized email is lower-cased and trimmed.
- Tenant foreign keys are explicit even when derivable, where they make RLS and indexing safer.
- Deletion is restricted or soft/archival by default. Hard deletion is reserved for approved privacy workflows and must preserve a non-identifying audit trail.
- Unique constraints are scoped to the owning organization unless a global uniqueness rule is explicitly stated.
- Sensitive fields are minimized; no passwords, access tokens, service-role keys, or full clinical records are stored.

## `profiles`

**Purpose:** Application identity linked one-to-one to Supabase Auth. **Primary key:** `id uuid`; also foreign key to `auth.users(id)` with restricted deletion. **Columns:** `id` required; `display_name` optional; `preferred_name` optional; `legal_name` unresolved; `status` required (`active`, `suspended`, `deactivated`); `created_at`, `updated_at`, `last_seen_at` optional. **Defaults:** UUID is supplied by Auth; status defaults to `active`; timestamps default to UTC now. **Constraints/indexes:** one row per Auth user; index status and lower-name search only if needed. **Ownership:** no organization owner; memberships own access. **Deletion:** deactivation is preferred; Auth deletion requires an approved profile-retention policy. **Audit/RLS:** users may read/update their safe profile fields; administrators cannot infer access from profile alone; profile status must participate in access checks. Sensitive: names and activity timestamps.

## `organizations`

**Purpose:** Customer tenant. **Primary key:** `id uuid`. **Columns:** `name` required; `slug` required and normalized; `status` required (`active`, `archived`, `deleted`); optional `legal_name`, `default_timezone`, `default_country_code`; timestamps and optional `archived_at`, `archived_by`. **Constraints/indexes:** unique slug; bounded non-empty name; index status. **Foreign keys:** archival actor to profiles if retained. **Ownership:** root tenant. **Deletion:** archive/tombstone; restrict hard deletion while children or audit references exist. **Audit/RLS:** active members read; owners/admins manage; creation is a controlled onboarding transaction.

## `clinics`

**Purpose:** Operating clinic within an organization. **Primary key:** `id uuid`. **Columns:** `organization_id` required; `name` required; optional `slug`, `status`, `timezone`, `archived_at`, `archived_by`, timestamps. **Constraints/indexes:** organization foreign key; unique `(organization_id, slug)` when slug exists; index `(organization_id, status)`. **Deletion:** archive before tombstone. **Tenant:** organization. **Audit/RLS:** active organization members can read active clinics subject to scope; organization owner/admin and explicitly permitted clinic administrators manage according to policy.

## `locations`

**Purpose:** Physical or virtual service location. **Primary key:** `id uuid`. **Columns:** `clinic_id` and denormalized `organization_id` required; `name` required; optional `address` fields, `timezone`, `status`, `archived_at`, `archived_by`, timestamps. **Constraints/indexes:** clinic must belong to organization; unique `(clinic_id, name)` for active records if approved; indexes `(organization_id, status)` and `(clinic_id, status)`. **Deletion:** archive/tombstone; never cascade silently into historical references. **Tenant:** organization through clinic. **Audit/RLS:** read and manage only within active membership scope.

## `organization_memberships`

**Purpose:** User-to-organization access relationship. **Primary key:** `id uuid`. **Columns:** `organization_id`, `profile_id`, `status` required, `invited_at`, `accepted_at`, `suspended_at`, `revoked_at`, optional reason fields, timestamps. **Constraints/indexes:** unique active or non-revoked membership per `(organization_id, profile_id)`; status transition checks; indexes `(profile_id, status)` and `(organization_id, status)`. **Deletion:** revoke rather than delete; retain security history. **Tenant:** organization. **Audit/RLS:** members may view permitted membership directory fields; inviting and status changes require appropriate organization permission; self-escalation is prohibited.

## `membership_clinic_scopes`

**Purpose:** Restricts a membership to listed clinics. **Primary key:** `id uuid` or composite unique key. **Columns:** `membership_id`, `organization_id`, `clinic_id`, timestamps, optional `created_by`. **Constraints/indexes:** unique membership/clinic pair; clinic organization must equal membership organization; index membership and clinic. **Deletion:** remove scope only through audited membership management. **Tenant:** organization. **RLS:** visible and mutable only to authorized membership administrators; scope rows never grant access by themselves without an active membership.

## `membership_location_scopes`

**Purpose:** Restricts a membership to listed locations. **Primary key:** `id uuid` or composite unique key. **Columns:** `membership_id`, `organization_id`, `clinic_id`, `location_id`, timestamps, optional `created_by`. **Constraints/indexes:** unique membership/location pair; location clinic and organization must match; indexes membership, clinic, and location. **Deletion/RLS:** same as clinic scopes; location scope is always bounded by its actual parentage.

## `roles`

**Purpose:** Named permission collection, either centrally defined system template or organization-owned custom role. **Primary key:** `id uuid`. **Columns:** optional `organization_id` for custom roles; `key` required; `name` required; `description` optional; `kind` required (`system`, `custom`); `status` required; optional `version`, timestamps, `archived_at`, `archived_by`. **Constraints/indexes:** system roles have null organization; custom roles require organization; unique `(organization_id, key)` for custom roles; system key globally unique. **Deletion:** archive; existing assignments retain historical role identity and become non-assignable. **RLS:** tenant users see roles available to their organization; system role mutation is platform-only.

## `permissions`

**Purpose:** Canonical permission catalogue. **Primary key:** `id uuid` or stable canonical key. **Columns:** `key` required, `description` required, `domain` required, `action` required, `status`, timestamps. **Constraints/indexes:** unique key; key must follow `domain.action` naming; no explicit deny rows initially. **Deletion:** deprecate, never silently reuse keys. **RLS:** readable to authenticated users as policy metadata; writable only by platform-controlled release processes.

## `role_permissions`

**Purpose:** Allow mapping from roles to permissions. **Primary key:** composite `(role_id, permission_id)` or UUID with unique pair. **Columns:** role, permission, timestamps, optional `created_by`. **Constraints/indexes:** unique pair; custom role and permission must be assignable; indexes by role and permission. **Deletion:** remove through audited role management; retain audit history. **RLS:** authorized role administrators only; permission catalogue remains platform-controlled.

## `membership_roles`

**Purpose:** Assign roles to memberships. **Primary key:** UUID or composite unique pair. **Columns:** `membership_id`, `role_id`, `organization_id`, timestamps, optional `assigned_by`, optional `expires_at`. **Constraints/indexes:** role must be system or belong to same organization; unique active assignment; index membership and role. **Deletion:** revoke assignment, do not rewrite historical audit data. **RLS:** active authorized organization administrators may manage; a user cannot assign a role they are not allowed to grant.

## `invitations`

**Purpose:** Pending or historical invitation to join an organization. **Primary key:** `id uuid`. **Columns:** organization, normalized email, optional target profile, status (`pending`, `accepted`, `expired`, `revoked`), opaque token hash, expiry, inviter, accepted profile, optional accepted time, timestamps. Scope and proposed role assignments belong in normalized invitation child records or a reviewed JSON contract; the first migration should choose one explicitly. **Constraints/indexes:** unique active invitation per organization/email; token hash unique; expiry required; no raw token. **Deletion:** revoke/expire; retain metadata without secrets. **RLS:** invitees see only their own safe pending invitation through a secure server path; authorized organization staff see safe invitation metadata.

## `audit_events`

**Purpose:** Append-oriented security and business-significant activity record. **Primary key:** UUID. **Columns:** actor profile nullable for system events; organization, clinic, location context nullable; action, entity type, entity UUID nullable, correlation/request ID, security classification, outcome, occurred_at, minimized before/after metadata, IP representation, user agent representation, retention marker. **Constraints/indexes:** action and entity bounded; context must be parent-consistent; no updates except controlled redaction; indexes by tenant/time, actor/time, correlation ID, and security class. **Deletion:** retention policy placeholder; any purge is controlled and audited. **RLS:** append through trusted server/security-definer path; normal clients cannot update or delete; readers require `audit.read` or platform authority. Sensitive: network metadata and change metadata; never secrets or full clinical records.

## Cross-table deletion and auditing

Parent archival must prevent new children and access. Foreign keys should use restrictive deletion unless a reviewed privacy workflow requires a tombstone. Sensitive operations include role assignment, invitation actions, membership status changes, scope changes, organization/clinic/location archival, and permission catalogue changes; each produces an audit event with minimized metadata.
